# Novel Receiving Review — 接收审稿反馈

> 借鉴 ECC receiving-code-review 的核心理念：**技术验证，不是盲目接受**。
> 审稿反馈是建议，不是命令。

## 核心原则

```
审稿反馈 = 建议 → 需要评估 → 不是盲目执行
```

---

## 处理流程

```
接收审稿反馈
      ↓
1. 阅读反馈（不反应）
      ↓
2. 理解需求（用自己的话复述）
      ↓
3. 验证问题（对照原文）
      ↓
4. 评估建议（适合本文？）
      ↓
5. 响应（认可或有理有据反驳）
      ↓
6. 修改（逐条执行）
```

---

## 禁止的回应

### ❌ 机械恭维

```
"您的意见太对了！"
"说得太好了！"
"非常感谢您的审稿！"
"谢谢您的建议！"
```

### ❌ 不经验证直接改

```
"好的，我现在就改"
（不检查原文就执行）
```

### ❌ 空泛反馈

```
"好的，我会改进的"
"明白了，我再写一遍"
（不具体指出问题）
```

---

## 正确的回应模式

### ✅ 认可时

```markdown
"明白了。行 45-47 的'眼中闪过'确实重复了三次，改为具体肢体语言。"
```

### ✅ 不认可时（有理由）

```markdown
"关于'主角降智推动剧情'的反馈，我认为：
- 第 78-82 行主角的决策是基于他当时掌握的信息
- 他的性格设定（谨慎型）决定了他不会冒进
- 如果需要修改，需要调整前文的信息披露节奏
建议：保持原文，或调整第 60-65 行的信息披露。"
```

### ✅ 不清楚时

```markdown
"关于'节奏失衡'的反馈：
- 我理解情节推进太快
- 但具体是哪个位置节奏突然加速？
- 是行 45-60 还是行 80-95？
请指出具体位置，我可以针对性地调整。"
```

## 问题分类与处理

### Critical 问题

```markdown
必须修复（致命问题）

识别标准：
- 人物性格崩塌
- 时间线矛盾
- 世界观设定违背
- 核心逻辑漏洞

处理：
- 立即修复
- 修复后重新提交审稿
```

### Important 问题

```markdown
建议修复（影响阅读体验）

识别标准：
- AI 套路化表达过多
- 对话不符合人物
- 钩子太弱
- 节奏明显失衡

处理：
- 评估是否合理
- 如果合理 → 修复
- 如果有异议 → 提出替代方案
```

### Suggestion 问题

```markdown
可选改进（锦上添花）

识别标准：
- 句式可以更多变
- 环境描写可以更具体
- 某个词可以替换

处理：
- 参考执行
- 不是必须
```

### Nit 问题

```markdown
细节问题（最低优先级）

识别标准：
- 个别词语
- 分段优化
- 标点调整

处理：
- 如果顺手就改
- 不改也可以
```

## 逐条处理清单

```markdown
收到审稿报告后：

□ 逐条阅读，不跳过
□ 逐条验证（对照原文）
□ 分类：Critical / Important / Suggestion / Nit
□ 优先处理 Critical
□ 处理时保持原文意思
□ 不确定的立即提问
□ 完成后报告修改情况
```

## 常见场景

### 场景 1：反馈模糊

```markdown
Reviewer: "这段节奏不好"

❌ 错误："好的，我改"

✅ 正确："'节奏不好'能具体指出是太快还是太慢吗？
是开场信息太多（第 10-30 行）还是高潮铺垫不够（第 80-100 行）？"
```

### 场景 2：反馈矛盾

```markdown
Reviewer A: "主角太软弱"
Reviewer B: "主角太强硬"

✅ 正确："两位审稿人对主角性格有不同理解。
建议确认：主角的性格定位是？
A. 成长型（目前软弱，逐步变强）
B. 内敛型（外表软弱，内心坚定）
C. 其他定位"
```

### 场景 3：反馈超出范围

```markdown
Reviewer: "建议把主角改成女性"

✅ 正确："这个方向性改动超出当前审稿范围。
是否需要用户（作者）确认主角性别变更？
如果确认，我需要同步更新人物设定和大纲。"
```

### 场景 4：多次返修后仍有问题

```markdown
第 2 次返修审稿仍不通过：

✅ 正确："本次审稿仍有问题，需要您介入决策：

问题：
- [问题1]
- [问题2]

我的判断：
- [我的理解]

建议：
A. 调整大纲（根本性改变）
B. 继续返修（我尝试第 3 次）
C. 暂时跳过本章（先写后面的情节）

请选择。"
```

## 修改执行顺序

```markdown
处理反馈时，按以下顺序：

1. Critical → 立即修复
2. Important（有共识）→ 修复
3. Important（有异议）→ 提出替代方案
4. Suggestion → 参考执行
5. Nit → 顺手就改

每条修改后：
- 标注修改位置
- 简单说明修改内容
```

---

## 禁止事项

- ❌ 机械恭维（"太好了"、"非常感谢"）
- ❌ 不验证就执行
- ❌ 跳过模糊问题不提问
- ❌ 盲目接受所有反馈
- ❌ 随意扩展反馈范围
- ❌ 修改后不报告
