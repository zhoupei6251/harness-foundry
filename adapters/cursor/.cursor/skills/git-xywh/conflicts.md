# 冲突相关陷阱

## 检测

- 二进制文件冲突 = Git 无法展示 diff — 需选择完整一侧版本
- 「双方修改」与「双方新增」冲突 — 预期解决方式不同
- 重命名且修改 = Git 可能未识别重命名 — 出现「假」冲突
- 若 diff 忽略空白，仅空白冲突可能被隐藏

## 解决

- 代码里忘记删 `<<<<<<<` 等标记 = 能编译但逻辑错误
- 该用「接受我方」却用了「接受对方」= 难以撤销
- 合并提交里冲突解决不当 = 静默引入缺陷
- 变基期间 `git checkout --ours` 与合并时语义**相反**

## 变基过程中

- 每个提交可能各有冲突 — 要解决 N 次，不是 1 次
- 变基时跳过提交 = 提交丢失且警告不明显
- 未全部解决就 `--continue` = 报错，不会出现「部分合并」
- 变基多步后中止 = 回到起点，中间工作丢失

## 工具问题

- 外部合并工具未保存 = Git 认为已解决但文件未变
- 工具删掉标记但未合并代码 = 静默错误
- `git mergetool` 可能生成 `.orig` 备份，易被误提交
- 三方合并工具在变基时「本地/远程/基准」含义易颠倒

## 预防

- 经常拉取，避免分歧过大
- 小而专注的提交 = 冲突面更小
- 与团队沟通正在改哪些文件
- 向 `develop` 合入前，在功能分支上与 **`origin/develop` 对齐** 后再解决冲突；也可用 `git diff develop`（或团队规定的基础分支）预览分歧
