# 协作相关陷阱（含受保护分支）

## 组织策略（与本仓库相关）

- **签名提交 / 签名标签** — 若团队启用 GPG 或 SSH 签名，以组织文档与托管平台设置为准。
- **克隆与凭据** — 使用组织允许的协议（HTTPS/SSH）；**勿**在脚本或 CI 日志中硬编码 token；凭据走各平台推荐的凭据助手或密钥管理。
- **不要直接向 `main`、`develop` 推送个人开发提交** — 使用 MR/PR；以远程分支保护配置为准。
- **`main` / `develop` 协作回滚** — 使用 **`git revert`**，不要用 **`git reset --hard`** 改写已发布历史。
- **误合并的 MR** — `git revert -m 1 <merge-commit-hash>`，再推送（见 `workflow.md`）。
- **公共受保护分支** — 不使用 `git push --force`；个人功能分支若允许强推，优先 `--force-with-lease`。

## 拉取 / 推送

- `git pull` = fetch + merge — 可能产生意外合并提交。
- `git pull --rebase` 或先 `fetch` 再 `rebase origin/develop` — 功能分支上常用，但仍可能冲突。
- 非快进被拒 ≠ 必须强推 — 应先与 `develop` 对齐。
- `--force` 会覆盖他人历史 — 即使在允许强推的分支也应优先 `--force-with-lease`。

## 强推

- `--force` 忽略他人更改 — 同事提交可能丢失。
- `--force-with-lease` 在远程有更新时会失败 — 更安全但非万能。
- 对 `main` 及受保护分支强推 = CI/CD 与审计风险。
- 平台分支保护可禁止强推 — 建议开启。

## 远程分支

- `git fetch` 不更新工作区 — 只更新引用。
- 远程重命名分支不会自动更新本地跟踪关系。
- `origin` 只是约定 — 可以有多个远程。
- `git remote prune origin` 清理远程已删引用，不会删本地分支。

## 代码评审

- 评审中又推送 = 新提交未必已被审到。
- 评审中强推 = diff 变化，评论可能失效。
- CI 未完成就批准 = 缺陷可能被合并。
- 压缩合并会丢失单次提交的独立历史。

## 团队协调

- 多人同分支 = 冲突不断。
- 无分支命名规范 = 长期项目混乱。
- 开工前忘记拉取 = 历史分叉。
- 未通知就变基共享分支 = 破坏他人工作。
