## 四、正确性审查

### 4.1 准确且与主流 Git 行为一致的内容

- **变基与合并中 `ours`/`theirs` 语义差异**（`conflicts.md`）表述正确，能减少常见误操作。
- **`revert` 对合并提交需 `-m`**（`workflow.md`、`history.md`、`commands.md`）一致。
- **`git pull` = fetch + merge** 可能产生合并提交（`collaboration.md`）正确；建议 `pull --rebase` 的说明亦合理。

### 4.2 需细化或易误导处

| 位置 | 内容 | 说明 |
|------|------|------|
| `advanced.md` 交互式变基 | `git rebase -i develop` | 通常等价于「对 `develop` 与当前分支分叉以来的提交」做 todo，但读者可能误以为「编辑 develop 上的提交」。可加半句注释：「在功能分支上执行，重写的是本分支相对 develop 的提交」。 |
| `advanced.md` reflog 恢复 | `git reset --hard HEAD@{5}` | 下标因 reflog 变化而不同，文档已给示例性质说明即可，若可加一句「以 `git reflog` 实际序号为准」更稳。 |
| `advanced.md` 恢复 stash | `git fsck --unreachable \| grep commit` | 在 **Windows CMD/PowerShell** 中管道与 `grep` 未必可用；技能元数据包含 `win32`，建议改为「在 Git Bash 中」或提供 `git fsck` + 人工查看的跨平台写法。 |
| `SKILL.md` 冲突检测 | `grep -r "<<<\|>>>\|===" .` | 在 Windows 原生环境可能无 `grep`；建议注明使用 Git Bash、或 `git grep`、或 IDE 检查（文中已有 IDE 备选，可把命令行方案标为「类 Unix / Git Bash」）。 |
| `commands.md` `git tag` 示例 | `git push origin v2.3.12` | 标签与分支同名时的歧义（`branching.md` 已提）在操作流程里仍可再提醒「避免与分支同名」。 |

### 4.3 版本与分支语义

- **Va.b.c.d** 与业界常见 **SemVer**（主.次.补丁）不完全同构，文档已说明「以发布负责人为准」——逻辑上自洽；若面向新人，可加一句「勿与 npm/package 版本号规则自动等同」。
