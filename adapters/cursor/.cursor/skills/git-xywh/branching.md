# 分支相关陷阱（结合本组织模型）

以下在通用 Git 行为之外，特别标注 **`main` / `test` / `develop` 与 `feature` / `task` / `temp` / `bugfix` / `test/v*`** 场景下的典型问题。总览见 `workflow.md`。

## 与公司模型相关的陷阱

- **从错的基础分支建功能分支** — 日常功能应从 **`develop`** 建 `feature`/`task`/`temp`；**热修**应从 **`main`** 建 `bugfix/*`。搞反会导致夹带未发布代码或漏合修复。
- **在 `develop` 上长期直接开发** — 应使用临时分支再 MR，否则评审粒度差、回滚困难。
- **`bugfix` 只合 `main` 不合 `develop`** — 后续从 `develop` 发版会再次引入已修缺陷。
- **`test/v*` 上堆新功能** — 提测线应**以修缺陷为主**，否则测试环境与版本范围失真。
- **大型 feature 子分支长期不合回主 feature** — 易产生巨大合并与冲突；应按团队节奏合并。

## 创建分支

- `git checkout -b feature/foo` 时若当前不是 `develop` = 基准可能错误。
- `git branch feature` 未切换 = 仍留在原分支 — 提交会进错分支。
- 分支名含空格在某些工具中静默失败。
- 分支名中的 `/`（如 `feature/x`）= 部分系统当作目录。

## 切换分支

- `git checkout 分支` 时有未提交更改 = 可能带到新分支 — 易混淆。
- `git switch` 更安全，但 `-f` 会无警告丢弃更改。
- 没有自动 stash — 已跟踪的更改会阻塞，未跟踪的易混在一起。
- 切换到子模块不同的分支 = 子模块可能仍停留在旧状态。

## 合并

- 快进合并不产生合并提交 — 历史线性但缺少合并语境。
- `--no-ff` 始终产生合并提交 — 功能合并有用，小修可能显噪。
- 长期分支一次大合并 = 巨型合并提交难审难回滚。
- 合并后删分支 = 若无标签可能留下孤儿提交。

## 变基

- 对已发布分支变基 = 历史不同 = 他人需强拉。
- 交互式变基操作不当可能丢提交 — 用 reflog 恢复。
- 变基中冲突：每个提交都要解决，不是只解决一次。
- 变基改变 SHA — CI/CD 里对旧提交的引用会断。

## 远程跟踪

- 首次需 `git push -u origin feature` — 不加 `-u` 不建立跟踪。
- 远程删分支不会自动删本地跟踪 — `git fetch --prune` 清理。
- 未配置上游就 `git pull` = 报错 — 用 `git branch --set-upstream-to`。
- 远程改名不会自动更新跟踪 — 需手动重配。

## 命名约定

- 两个远程同名分支（`origin/main`、`upstream/main`）= 易混。
- Mac/Windows 分支名大小写不敏感，Linux 敏感 — 易致 CI 问题。
- 分支名与标签同名 = 部分命令歧义。
